### A-2: 중복 탐색 -- O(n^2)에서 O(n)으로 (15분)

#### 진행 방법

1. **간단 소개 (3분):**
   > "`examples/a2_find_duplicate.py`를 열어보세요. 문제는 간단합니다: 정수 배열에 중복이 있는지 판별하세요. 두 가지 방법이 있습니다. 첫 번째는 이중 for 루프로 모든 쌍을 비교하는 O(n^2)입니다. 두 번째는 해시 셋으로 이미 본 값을 추적하는 O(n)입니다. 둘 다 구현하고 측정해 보세요."

2. **학생 실습 (9분):**
   - 이중 루프를 사용한 O(n^2) 브루트포스 풀이를 구현합니다.
   - Python `set`을 사용한 O(n) 풀이를 구현합니다.
   - A-1에서 만든 타이머 유틸리티로 N=100, 1,000, 10,000, 50,000에서 실행 시간을 비교합니다.
   - 결과를 표로 정리하도록 안내합니다.

3. **체크포인트 (3분):**
   > "N=100에서는 둘 다 빠릅니다. 하지만 N=50,000에서는 O(n^2) 버전이 눈에 띄게 느려집니다 -- 수 초가 걸릴 수 있습니다 -- 반면 셋 기반 버전은 즉시 끝납니다. Big-O가 실제로 보이는 순간입니다. 강의에서 다룬 이론적 차이가 측정 가능한 실제 차이가 되었습니다."

---

### A-3: 실행 시간 그래프 (10분)

#### 진행 방법

1. **간단 소개 (2분):**
   > "이제 복잡도를 시각화해 봅시다. `examples/a3_complexity_plot.py`를 실행하세요. 이 스크립트는 다양한 입력 크기에서 더미 작업을 실행하고 matplotlib로 O(1), O(n), O(n log n), O(n^2) 그래프를 그립니다."

2. **학생 실습 (6분):**
   - 스크립트를 실행하고 생성된 그래프를 관찰합니다.
   - 그래프 모양만 보고 어떤 곡선이 어떤 복잡도인지 맞춰보도록 합니다.
   - 선택 사항: O(n^3) 곡선을 추가하거나 N 범위를 조정해 봅니다.

3. **체크포인트 (2분):**
   > "O(n^2)은 급격히 위로 치솟고, O(n)과 O(n log n)은 작은 스케일에서 비교적 평평합니다. 시각화를 보면 알고리즘 선택이 왜 중요한지 명확해집니다 -- N이 커질수록 더 심해집니다. 복잡도를 분석할 때 이 그래프를 떠올리세요."

## Type B -- 웹 코드 분석

### B-1: 상품 검색 API 비교 (15분)

#### 진행 방법

1. **간단 소개 (3분):**
   > "이제 복잡도가 실제 웹 애플리케이션에 어떤 영향을 주는지 살펴봅시다. `examples/b1_web_search/`에 Flask 앱이 있고 두 개의 검색 엔드포인트가 있습니다. 하나는 모든 상품을 순차 탐색하는 선형 검색이고, 다른 하나는 정렬된 리스트에서 이진 검색을 합니다. 서버를 시작하고 두 엔드포인트를 테스트해 보세요."

   서버 시작 방법을 안내합니다:
   ```
   cd examples/b1_web_search
   pip install flask
   python app.py
   ```

2. **학생 실습 (9분):**
   - Flask 서버를 시작하고 두 엔드포인트에 쿼리합니다:
     - `GET /search/linear?q=product_name` -- 선형 검색 O(n)
     - `GET /search/binary?q=product_name` -- 이진 검색 O(log n)
   - N=100, 10,000, 1,000,000에서 응답 시간을 측정하고 기록합니다.
   - 토론: N이 커질수록 응답 시간 차이는 어떻게 변하는가?

3. **체크포인트 (3분):**
   > "100개일 때는 두 엔드포인트가 거의 같은 속도입니다. 1,000,000개일 때는 선형 검색이 눈에 띄게 느려지는 반면 이진 검색은 거의 즉시 응답합니다. 이것이 정확히 O(n)과 O(log n)의 차이입니다. 실제 서비스에서 이 차이는 빠른 앱과 답답한 사용자 경험의 차이가 됩니다."
