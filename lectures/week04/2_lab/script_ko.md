## Type A -- 알고리즘 구현

### A-1: 병합 정렬 추적 (10분)

#### 진행 방법

1. **간단 소개 (2분):**
   > "`examples/a1_merge_sort_trace.py`를 열어보세요. 병합 정렬을 구현하는 것이 아닙니다 -- 지난 주에 이미 했습니다. 이번에는 재귀 호출 트리를 출력하여 배열이 어떻게 나뉘고 각 단계에서 어떻게 합쳐지는지 정확히 볼 수 있습니다. 실행하고 출력을 분석해 보세요."

2. **학생 실습 (6분):**
   - 제공된 샘플 배열로 스크립트를 실행합니다.
   - 출력을 관찰합니다: 각 재귀 호출이 처리 중인 부분 배열을 보여줍니다.
   - 재귀의 깊이를 추적하고 병합 연산의 횟수를 셉니다.
   - 다양한 입력 배열(정렬된, 역순, 랜덤)로 호출 트리 구조가 달라지는지 확인하도록 권장합니다.

3. **체크포인트 (2분):**
   > "호출 트리는 입력에 관계없이 항상 log n 단계입니다. 각 단계에서 모든 원소가 정확히 한 번의 병합에 참여합니다. 따라서 단계당 n의 작업 곱하기 log n 단계 -- 총 O(n log n)입니다. 이 구조를 눈으로 보면 복잡도 분석이 구체적으로 이해됩니다."

### A-2: k번째로 작은 원소 찾기 (15분)

#### 진행 방법

1. **간단 소개 (3분):**
   > "`examples/a2_kth_smallest.py`를 열어보세요. 문제: 정렬되지 않은 배열에서 k번째로 작은 원소를 찾으세요. 배열을 O(n log n)에 정렬할 수도 있지만 더 나은 방법이 있습니다. Randomized Select 알고리즘은 퀵 정렬의 분할 단계를 사용하여 피벗의 한쪽으로 탐색을 좁힙니다. 평균적으로 O(n) 시간에 동작합니다. 알고리즘을 구현해 보세요."

2. **학생 실습 (10분):**
   - 분할(partition) 함수를 구현합니다 (퀵 정렬에서 재사용 가능).
   - Randomized Select를 구현합니다: 랜덤 피벗 선택, 분할, 적절한 쪽으로 재귀.
   - 핵심 통찰: 퀵 정렬과 달리 한쪽에만 재귀합니다 -- 이것이 평균 O(n log n)이 아닌 O(n)인 이유입니다.
   - 예시로 테스트: 1번째로 작은 값(최솟값), n번째로 작은 값(최댓값), 중앙값 찾기.

3. **체크포인트 (2분):**
   > "핵심은 k번째 원소가 있을 쪽에만 재귀한다는 것입니다. 피벗이 위치 p에 놓이면, k가 p와 같으면 끝입니다. k가 p보다 작으면 왼쪽에, 크면 오른쪽에 재귀합니다. 평균적으로 문제가 절반씩 줄어들어 기대 O(n) 시간이 됩니다 -- 정렬보다 훨씬 빠릅니다."

---

### A-3: 최근접 점 쌍 (10분)

#### 진행 방법

1. **간단 소개 (2분):**
   > "`examples/a3_closest_pair.py`를 열어보세요. 2D 평면의 n개 점에서 거리가 가장 가까운 쌍을 찾는 문제입니다. 브루트포스는 모든 쌍을 확인 -- O(n^2)입니다. 분할 정복은 x좌표로 점을 나누고 각 반을 재귀적으로 풀고, 경계 스트립을 처리합니다. O(n log n)을 달성합니다. 코드에 두 구현이 모두 있으니 실행하고 비교해 보세요."

2. **학생 실습 (6분):**
   - 브루트포스와 분할 정복 버전을 모두 실행합니다.
   - 다양한 N 값(100, 1,000, 10,000)에서 실행 시간을 비교합니다.
   - 경계 스트립 로직을 분석합니다: 분할선에서 거리 d 이내의 점만 교차 비교가 필요합니다.
   - 개념적으로 가장 어려운 부분이므로 돌아다니며 막힌 학생을 도와줍니다.

3. **체크포인트 (2분):**
   > "브루트포스는 직관적이지만 느립니다. 분할 정복 버전은 우아합니다: 나누고, 각 반을 풀고, 경계 근처의 점만 확인합니다. 경계 스트립이 핵심입니다 -- 스트립 내 각 점에 대해 최대 6개의 점만 확인하면 됩니다. 이 최적화 없이는 결합 단계가 O(n^2)이 되어 아무것도 얻지 못합니다."

## Type B -- 웹 코드 분석

### B-1: 자동완성 API (15분)

#### 진행 방법

1. **간단 소개 (3분):**
   > "자동완성은 매일 사용하는 기능입니다 -- 몇 글자만 입력하면 제안이 나옵니다. `examples/b1_web_autocomplete/`의 Flask 앱을 열어보세요. 100,000개의 단어 사전이 있고 두 가지 검색 전략이 있습니다. 첫 번째는 모든 단어를 순차 탐색합니다. 두 번째는 정렬된 배열에서 이진 검색으로 접두사 범위를 찾습니다. 서버를 시작하고 글자를 입력하며 차이를 느껴보세요."

   서버 시작 방법을 안내합니다:
   ```
   cd examples/b1_web_autocomplete
   python app.py
   ```

2. **학생 실습 (9분):**
   - Flask 서버를 시작하고 두 엔드포인트에 쿼리합니다:
     - `GET /autocomplete/linear?q=pre` -- 모든 단어를 순차 검색
     - `GET /autocomplete/binary?q=pre` -- 정렬된 배열 + 접두사 범위 이진 검색
   - 점점 긴 접두사를 입력하며(예: "p", "pr", "pre", "pres") 응답 시간을 비교합니다.
   - 선형 방식은 느려지지만 이진 검색은 일관되게 빠른 것을 관찰합니다.
   - 토론: 정렬된 배열에서 이진 검색으로 접두사 매칭이 가능한 이유는 무엇인가요?

3. **체크포인트 (3분):**
   > "정렬된 사전에서 'pre'로 시작하는 모든 단어는 연속되어 있습니다. 이진 검색은 그 접두사 범위의 처음과 끝 위치를 O(log n) 시간에 찾습니다. 선형 방식은 매번 100,000개의 단어를 전부 탐색해야 합니다 -- O(n). 매 키 입력마다 실행되는 자동완성에서는 이 차이가 결정적입니다. 이것이 분할 정복의 실전 적용입니다: 이진 검색이 탐색 공간을 반복적으로 절반으로 줄입니다."
